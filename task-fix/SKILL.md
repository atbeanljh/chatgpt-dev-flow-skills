# 修复阶段（Bug Fix）— 独立于 require/design/plan/code 工作流

用于定位和修复项目中的 Bug 或问题。产出 `.codex/fix/fix-YYYYMMDD-xxx.md` 修复记录文件。

**重要**：本 skill 独立于 require/design/plan/code 四阶段工作流，不共用 `.codex/tasks/` 目录，不生成需求/设计/计划文档。

修复记录存放在项目根目录下的 `.codex/fix/` 目录中，每个修复会话为单独的文件 `fix-YYYYMMDD-xxx.md`。

工作流：`问题收集 → 根因分析 → 修复方案 → 编码修复 → 验证交付 → 自动收尾`

## 文件结构

```
<project-root>/.codex/fix/
  ├── fix-20260715-xxx.md          # 修复记录
  ├── fix-20260717-xxx.md          # 另一个修复会话
  └── ...
```

每个 `fix-*.md` 可以包含用户一次提交的**多个 Bug**（如 bug1、bug2），各自独立记录与追踪。

## 工作流

### 0. 自动确认上一个步骤

每次开始新的修复步骤时（如从「收集问题」进入「根因分析」），自动确认上一个步骤已完成：
- 更新修复文件中的会话记录，标记载上一个操作已完成
- 更新修复进度总览表

### 1. 创建或定位修复记录文件

**首次修复**：如果还没有 `.codex/fix/` 目录或当前会话对应的修复文件，运行：

```
python3 scripts/init_fix_folder.py <project-root> "<问题简述>"
```

捕获输出路径，得到修复记录文件。

**已有修复记录**：如果用户已有或指定了修复文件，直接读取使用，**不要**重新创建。

### 2. 收集与理解问题

与用户沟通，充分理解每个 Bug 的表现：

- **现象**：什么表现？如何复现？
- **环境**：哪个环境/版本？
- **影响范围**：影响多少用户/模块？
- **紧急程度**：P0/P1/P2？

> 如果用户描述的 Bug 不够清晰，**主动向用户询问**以补充信息（如复现步骤、日志、堆栈等），然后写入修复文件。
> 将每个 Bug 写入修复文件的「Bug 列表」中，分配 BUG-001、BUG-002 ... 编号。

### 3. 根因分析

深入阅读相关源码、数据流、调用链：
- 从现象出发，沿调用链路逆向追踪
- 检查数据流、状态变更、边界条件
- 确认是逻辑错误、数据问题、配置错误还是兼容性问题
- 将根因分析结果写入对应 Bug 的「修复方案」字段

### 4. 设计修复方案

基于根因设计每个 Bug 的修复方案，写入对应 Bug 的「修复方案」字段。
明确变更范围、风险和验证方法。

### 5. 编码修复

按修复方案逐一实现代码：

- 最小变更原则——只改必须改的代码
- 每个变更后做最小验证（编译通过）
- 每次关键进展后更新修复文件中的会话记录和修复进度总览
- 一个 Bug 修复并验证通过后，标记 `[x]`、更新状态为"已验证"、填入完成时间
- 再进入下一个 Bug 的修复

### 6. 验证、标记与交付

- 编译验证
- 运行相关测试
- 向用户说明：修复了哪些 Bug、根因是什么、验证结果
- 更新修复文件头部的「状态」为"全部完成"
- 更新修复进度总览表

### 7. 自动收尾

所有 Bug 修复完成后，自动执行收尾工作：

- **最终确认**：确认所有 Bug 前的 `[ ]` 均已标记为 `[x]`、所有 Bug 状态为"已验证"
- **更新修复文件**：更新修复文件头部「状态」为"全部完成"
- **更新状态文件**：更新 `status.md` 标记修复完成（fix_status → ✅ 已完成）
- **整体总结**：向用户提供完整的修复总结——修复了哪些 Bug、根因是什么、验证结果、涉及的文件变更

## fix-*.md 文档结构

每个修复文件包含**所有用户这次提交的 Bug**。结构如下：

```markdown
# 修复记录

**创建时间**：YYYY-MM-DD HH:mm
**状态**：收集中 / 修复中 / 全部完成
**修复会话**：fix-YYYYMMDD-xxx

---

## Bug 列表

### BUG-001

- [x] <!-- bug_id: BUG-001 / 提出时间: YYYY-MM-DD HH:mm -->
  **用户反馈**: <用户描述的原始 Bug 现象>
  **修复建议**: <初步的修复方向>
  **修复方案**: <根因分析和具体修复方案>
  **修复状态**: 已验证
  **验证结果**: 编译通过 / 测试通过 / 手动验证通过

### BUG-002

- [ ] <!-- bug_id: BUG-002 / 提出时间: YYYY-MM-DD HH:mm -->
  **用户反馈**: <用户描述的原始 Bug 现象>
  **修复建议**: 待分析
  **修复方案**: 待分析
  **修复状态**: 待修复
  **验证结果**: -

---

## 修复进度总览

| Bug ID | 状态 | 完成时间 |
|--------|------|----------|
| BUG-001 | ✅ 已完成 | YYYY-MM-DD HH:mm |
| BUG-002 | ⏳ 待修复 | - |

## 会话记录

| 时间 | 操作 | 备注 |
|------|------|------|
| HH:mm | 创建修复会话 | - |
| HH:mm | BUG-001 根因分析完成 | .... |
| HH:mm | BUG-001 编码修复完成 | .... |

## 恢复标记

last_processed_bug: BUG-002
last_action: 待修复
```

## Bug 编码规则

- Bug ID 从 `BUG-001` 开始依次递增，不重复
- 同一批提交的多个 Bug 依次编号：BUG-001、BUG-002、BUG-003 ...
- 每个 Bug 独占一个三级标题 `### BUG-NNN`
- 标题下紧跟一个 `- [ ]` 勾选条目，勾选表示该 Bug 已修复并验证完成
- `<!-- ... -->` 注释中记录 bug_id 和提出时间，供脚本自动解析

## status.md 跟踪字段

| 字段 | 更新时机 | 示例值 |
|------|----------|--------|
| `fix_status` | 全部修复完成后 | ✅ 已完成 |
| `fix_artifact` | 修复完成后 | `.codex/fix/fix-20260717-xxx.md` |
| `current_phase` | 状态变更 | 全部完成 |
| `progress_note` | 每次关键操作 | BUG-002 根因分析完成，待编码 |
| `todos` | 每次关键操作 | 1. 修复 BUG-002... |

## 关键原则

1. **独立目录**：所有修复记录放在 `.codex/fix/` 目录下，不与 require/design/plan/code 共用 `.codex/tasks/`
2. **单文件多 Bug**：一个修复文件可以包含用户一次提交的多个 Bug，各自独立追踪
3. **不懂就问**：如果 Bug 描述不清，主动向用户询问，不要在信息不全时自行猜测
4. **勾选标记**：每个 Bug 前的 `[ ]` → `[x]` 表示该 Bug 已修复且验证通过
5. **最小变更**：只修 Bug，不做无关重构
6. **全面验证**：修复后必须确认能正常运行（编译 + 测试 + 手动验证）
7. **根因优先**：不要只修表面症状

## 中断恢复机制

由于修复工作可能跨多个任务中断（terminal 断开、上下文耗尽等），通过以下机制确保可恢复：

1. **修复文件中的「恢复标记」**：文件底部 `last_processed_bug` 和 `last_action` 记录最后处理到的 Bug 和执行的操作
2. **会话记录表**：时间线记录每一步关键操作
3. **进度总览表**：快速查看哪些 Bug 已完成、哪些待处理
4. **恢复流程**：
   - 读取 `.codex/fix/` 目录，找到最近或用户指定的修复文件
   - 读取「恢复标记」和「进度总览」了解中断位置
   - 从标记的 `last_processed_bug` 接续工作
   - 如果文件头部「状态」为"全部完成"，询问用户是否需要新会话
