# 需求分析阶段

分析用户原始需求，产出一份符合 `.cospec` 风格的需求文档 `requirements.md`。
本阶段是四个阶段的**第一步**：
`需求分析 → 方案设计 → 实施计划 → 编码实现`
由用户通过 `/design`、`/plan`、`/code` 等触发词手动调用下一阶段。

## status.md 说明

status.md 由工作流在执行过程中自动维护，记录阶段状态和进度。

### 更新字段速查

| 字段 | 更新时机 | 示例值 |
|------|----------|--------|
| `req_status` | 需求确认后 | ✅ 已完成 |
| `req_time` | 需求确认后 | 2026-07-16 14:30 |
| `req_artifact` | 需求确认后 | `01-requirements.md` |
| `current_phase` | 确认后切换 | 方案设计 |
| `progress_note` | 每次关键操作 | 分析完成，待用户确认 |
| `todos` | 每次关键操作 | 1. 确认需求文档 |

### 中断与恢复

- 进度通过 `progress_note` 和 `todos` 记录，阶段位置通过 `current_phase` 追踪
- 恢复时读取 status.md 跳过已完成阶段，从当前位置继续

## 工作流

### 1. 创建或定位任务文件夹

**用户提供任务标题时（首次）**：运行 `scripts/init_task_folder.py <project-root> "<task-title>"` 创建文件夹，捕获输出的文件夹路径。

**中断恢复场景**：运行 `scripts/resolve_task_folder.py <project-root>` 获取最近任务文件夹，读取 `status.md` 确定中断位置后继续。

**用户指定了路径时**：直接使用。

### 2. 读取 status.md（断点恢复）

读取 `status.md`，检查当前阶段和进度。如果"需求分析"已完成，询问用户是重新分析还是进入下一阶段。

### 3. 全面分析项目结构

阅读源码，关注：模块划分、相关代码、数据模型、API 接口、编码惯例、配置方式。
将要点记入 `status.md` 的 `进度说明`。

### 4. 澄清需求

与用户沟通边界和验收标准，直至消除歧义。
每次交互后更新 `status.md` 的进度说明和待办事项。

### 5. 编写需求文档

使用下方文档结构写入 `01-requirements.md`。

### 6. 确认阶段完成

将文档给用户确认。
- **用户确认**：更新文档状态为"已确认"，更新 `status.md` 标记需求分析完成并切换到方案设计阶段，告知用户用 `/design` 进入下一阶段。
- **用户要求修改**：按反馈修改，更新 `status.md` 进度说明，循环直到确认。

## 文档结构

```markdown
# 需求文档

**创建时间**：YYYY-MM-DD
**状态**：需求确认中 / 已确认

## 用户故事

### 故事 1 - <标题>（优先级：P<n>）

<角色>在<场景>下，需要<做什么>，以便<带来什么价值>。

**验收场景**：

1. **条件**：<前置条件>，**操作**：<用户操作>，**预期结果**：<期望行为>
2. **条件**：<前置条件>，**操作**：<用户操作>，**预期结果**：<期望行为>

**边界情况**：

- **条件**：<边界条件>，**操作**：<用户操作>，**预期结果**：<期望行为>

### 故事 2 - <标题>（优先级：P<n>）
...
```

## 查找已有任务文件夹

如果用户引用已有任务但没有提供路径，运行 `scripts/resolve_task_folder.py <project-root>` 选择最近的任务文件夹。 

## 资源

- `scripts/init_task_folder.py`: 创建任务文件夹和初始 status.md。
- `scripts/resolve_task_folder.py`: 从显式输入或最新文件夹解析任务文件夹路径。
