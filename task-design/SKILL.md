# 方案设计阶段（功能 + 技术）

将原 task-design 和 task-tech 合并为一个阶段。基于已确认的 `requirements.md` 产出 `design.md`，包含功能设计和技术设计两部分。

本阶段是四个阶段的**第二步**：
`需求分析 → 方案设计 → 实施计划 → 编码实现`
由用户通过 `/plan`、`/code` 等触发词手动调用下一阶段。

## 前置条件

- 任务文件夹中必须有已确认的 `requirements.md`（状态：已确认）
- 如果状态仍为"需求确认中"，不推进 — 先让用户确认
- 如果 status.md 中 design_status 为"✅ 已完成"，询问是重新设计还是直接进入下一步

## 工作流

### 0. 自动确认上一阶段

启动方案设计时，自动检查并确认上一阶段（需求分析）：
- 如果 `status.md` 中 `req_status` 不是"✅ 已完成"，自动将其更新为"✅ 已完成"
- 更新 `req_time` 为当前时间
- 在进度说明中记录"需求分析已自动确认"
- 向用户说明：需求分析阶段已完成、已自动确认，现进入方案设计

### 1. 定位任务文件夹

运行 `scripts/resolve_task_folder.py <project-root>` 或使用用户提供的路径。
读取 `status.md` 检查当前阶段和进度。

### 2. 全面理解需求与代码库

加载 `requirements.md`，理解用户故事和验收场景。
深入分析项目源码：
- 构建系统和依赖配置
- 现有类层次和包组织
- 编码惯例和框架
- 数据库 schema 和迁移脚本
- 测试基础设施

### 3. 输出设计文档

使用下方文档结构写入任务文件夹，文件名 `02-design.md`（含编号前缀以保持阶段排序）。
文档涵盖功能设计和技术设计两部分。

### 4. 确认阶段完成

将文档呈现给用户确认。
- **用户确认**：更新文档状态为"已确认"，更新 `status.md` 标记方案设计完成并切换到实施计划阶段，告知用户用 `/plan` 进入下一阶段。
- **用户要求修改**：按反馈修改，更新 `status.md` 进度说明，循环直到确认。

## status.md 更新说明

status.md 由工作流在执行过程中自动维护。

### 更新字段速查
| 字段 | 更新时机 | 示例值 |
|------|----------|--------|
| `design_status` | 设计确认后 | ✅ 已完成 |
| `design_time` | 设计确认后 | 2026-07-16 14:30 |
| `design_artifact` | 设计确认后 | `02-design.md` |
| `current_phase` | 确认后切换 | 实施计划 |
| `progress_note` | 每次关键操作 | 设计完成，待用户确认 |
| `todos` | 每次关键操作 | 1. 确认设计文档... |

### 中断与恢复
- 进度通过 `progress_note` 和 `todos` 记录，阶段位置通过 `current_phase` 追踪
- 恢复时读取 status.md 跳过已完成阶段，从断点继续
